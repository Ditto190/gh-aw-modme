# ADR-40837: Add `wgdonenotdeferred` linter for non-deferred `sync.WaitGroup.Done()` calls

**Date**: 2026-06-22
**Status**: Draft

## Context

The codebase maintains a custom `go/analysis` linter suite (`pkg/linters/*`, registered in `cmd/linters/main.go`) to enforce house style and catch concurrency anti-patterns that off-the-shelf golangci-lint rules miss. When `wg.Done()` is called directly rather than via `defer`, a panic or early return before that call leaves the `WaitGroup` counter undecremented, and any goroutine blocked on `wg.Wait()` deadlocks forever. The existing `manualmutexunlock` and `contextcancelnotdeferred` analyzers already enforce the same defer-cleanup idiom, and both current `wg.Done()` usages in the repo (`pkg/console/spinner.go`, `pkg/cli/mcp_inspect_inspector.go`) already use `defer`, so a new rule must fit the established framework while producing zero false positives on existing code.

## Decision

We will add a new `go/analysis` analyzer package `pkg/linters/wgdonenotdeferred` and register it in `cmd/linters/main.go`. The analyzer visits every `*ast.FuncDecl` and `*ast.FuncLit` body independently (declining to descend into nested `FuncLit`s so closures are not double-reported), and flags any statement-level `wg.Done()` call appearing as an `*ast.ExprStmt` — a deferred call parses as a `DeferStmt`, so checking for `ExprStmt` naturally excludes correctly-deferred calls. The receiver type is resolved via `pass.TypesInfo` and the diagnostic fires only when the static type is `sync.WaitGroup` or `*sync.WaitGroup`. Test files are skipped via `filecheck.IsTestFile`, and `//nolint:wgdonenotdeferred` directives are honored via the shared `nolint` helper.

## Alternatives Considered

### Alternative 1: Rely on an off-the-shelf linter

No enabled golangci-lint rule flags non-deferred `WaitGroup.Done()`. Adopting an external linter (or re-enabling a disabled one) to cover this would pull in unrelated checks and is subject to upstream timelines outside our control. Rejected in favor of a focused local analyzer consistent with the existing `pkg/linters` suite.

### Alternative 2: Name-based matching on `Done()` without type resolution

The analyzer could flag any selector call ending in `.Done()` without consulting type information. This is simpler but would misreport unrelated `Done()` methods (e.g. `context.Context.Done()`, or any user type with a `Done()` method — exercised by the `Finisher` fixture). Rejected; static type resolution against `sync.WaitGroup` keeps false positives at zero.

## Consequences

### Positive

- Catches a real deadlock anti-pattern that no currently enabled linter detects, extending the defer-cleanup family alongside `manualmutexunlock` and `contextcancelnotdeferred`.
- Type-resolved matching plus per-function-body traversal yield zero false positives on current code (both existing `wg.Done()` sites already defer) and avoid duplicate diagnostics for nested closures.
- Follows the established analyzer pattern, so it is consistent with the rest of the suite and covered by `analysistest`-based tests (goroutine, helper-function, and value-receiver bad cases; deferred good cases; unrelated `Done()` exemption).

### Negative

- Adds another analyzer to maintain and run, marginally increasing lint time and the registration surface in `cmd/linters/main.go`.
- Only direct statement-level `wg.Done()` calls are analyzed; a `Done()` reached through indirection (function value, interface, or wrapper) is not in scope.

### Neutral

- The rule is purely diagnostic (`pass.ReportRangef`); it offers no automated fix/suggested edit.
- Authors who intentionally call `Done()` without `defer` can suppress with `//nolint:wgdonenotdeferred`.
